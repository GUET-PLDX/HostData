# User — Application Layer

## OVERVIEW

Hardware-to-framework bridge and YAML-based robot configuration. Entry point for all application logic.

## STRUCTURE

```
User/
├── app_main.cpp        # C++ entry: peripheral instantiation + HardwareContainer
├── app_main.h          # C linkage header for app_main()
├── xrobot_main.hpp     # Generated: module instantiation from YAML, main loop
├── xrobot.yaml         # Default/minimal robot config (BlinkLED only)
├── libxr_config.yaml   # LibXR framework settings
├── flash_map.hpp       # Auto-generated flash sector table (STM32F407)
└── RobotConfig/
    ├── sentry_gimbal.yaml   # Gimbal board, launcher, host/autosight input
    └── sentry_chassis.yaml  # Chassis board, referee and power control
```

## WHERE TO LOOK

| Task | Location | Notes |
|------|----------|-------|
| Map new STM32 peripheral | `app_main.cpp` | Add `STM32*` wrapper + `LibXR::Entry` in `HardwareContainer` |
| Create new robot config | `RobotConfig/<name>.yaml` | Define modules, PID params, motor assignments |
| Change default config | `xrobot.yaml` | Used when no `-c` flag passed to build.sh |
| Modify terminal/debug | `app_main.cpp` | `STDIO::read_`/`write_` + Terminal setup |
| Flash storage layout | `flash_map.hpp` | Auto-generated, matches `DevC.ioc` MCU |

## KEY PATTERNS

### app_main.cpp Structure
1. **HAL externs**: `extern CAN_HandleTypeDef hcan1;` etc.
2. **DMA buffers**: Static arrays for SPI/UART/USB transfers
3. **Hardware instantiation**: `STM32CAN`, `STM32UART`, `STM32SPI`, etc.
4. **USB setup**: CDC-UART over both OTG-FS and OTG-HS
5. **HardwareContainer**: String-named registry mapping logical names to physical objects
6. **Terminal**: USB CDC as stdio, RamFS-backed debug shell
7. **Database**: Flash-backed persistent storage
8. **`XRobotMain(peripherals)`**: Hand-off to generated robot code

### YAML Config Syntax
```yaml
modules:
- id: motor_yaw                    # Instance ID (referenced by @&motor_yaw)
  name: RMMotor                    # Module class name
  constructor_args:                # Maps to C++ constructor params
    param:
      model: RMMotor::Model::MOTOR_GM6020
      feedback_id: 522
      can_bus_name: can1           # Matches HardwareContainer string name
  template_args:                   # Optional: C++ template parameters
    ChassisType: Helm
```

### Hardware Name Mapping
Logical names in `HardwareContainer` (e.g., `"can1"`, `"uart_dr16"`, `"spi_bmi088"`) are referenced by module YAML configs via `can_bus_name`, `uart_name`, etc. Adding a peripheral requires both:
1. C++ instantiation in `app_main.cpp`
2. `LibXR::Entry` registration with logical name(s)

## ANTI-PATTERNS

- **DO NOT** edit `xrobot_main.hpp` manually — it is regenerated by `xrobot_gen_main`
- **DO NOT** edit `flash_map.hpp` manually — auto-generated from `.ioc`
- **DO NOT** break logical name strings — modules depend on exact matches
- `NOLINTBEGIN`/`NOLINTEND` and `clang-format off/on` regions in `app_main.cpp` are intentional for the hardware declaration block
